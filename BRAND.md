# pintr ‚Äî brand spec

The single source of truth for pintr's identity. The SVG paths live in code, not
here: the mark + wordmark in `src/components/wordmark.tsx`, the icon renditions in
`scripts/generate-icons.mjs`. This doc defines the rules.

## The mark (pin = pint)

A map-pin teardrop with a neon-pink rim, a frothy pint inside (amber beer + foam
head), tapering to a point that matches the pin. Two renditions:

- **App icon:** on a full navy tile, **with the fading pink halo**, gradient fills.
  Rich rendition ‚Äî this is where the glow belongs.
- **UI mark** (`<Mark>`): the same pin, solid fills, transparent background, used
  inline next to the wordmark. Halo kept (subtle at small size).

**Never change the pin geometry** without re-approving ‚Äî it's settled.

## The wordmark

Lowercase `pintr`, heavy geometric sans (Geist), tight tracking. **Clean-cut ‚Äî
never a glow behind the type** (a logo must read flat/tiny/mono; glows muddy
letterforms and date). `pint` in **amber**, `r` in **pink**.

- Default: amber `pint` + pink `r` on dark.
- **Light variant** (`tone="light"`): foam-white `pint` + pink `r` ‚Äî use on amber
  fills or any surface where amber lacks contrast.
- The optional pink underline is reserved; default is no underline.

## App icon rules (from the icon-standards research)

- **Full-bleed square master, no pre-rounded corners, no transparency** (iOS +
  maskable apply their own mask).
- **Maskable variant:** mark scaled to ~70% and centred so it (and its halo) sit
  inside the **80% safe zone** (centre circle, radius 40% of width).
- `any`/favicon: fuller mark is fine (rendered unmasked).
- Generated by `npm run icons`.

## Do / don't

- ‚úÖ Glow on the app icon. ‚úÖ Clean wordmark everywhere else. ‚úÖ Amber = action,
  pink = accent/ping.
- ‚ùå No glow behind type. ‚ùå Don't re-round icon corners. ‚ùå Don't let pink become
  a second primary ‚Äî it's a spark, used sparingly.
